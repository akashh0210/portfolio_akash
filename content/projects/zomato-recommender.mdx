## Problem

Standard keyword search for restaurant discovery returns low-quality results for natural language queries like "somewhere quiet for a work lunch with decent vegetarian options near Indiranagar." The Zomato dataset is rich — cuisine, cost, ratings, locality — but keyword search ignores most of it.

## Hypothesis

A lightweight embedding + LLM re-ranking pipeline on top of the structured Zomato dataset would return meaningfully better results for natural-language queries without requiring a large-scale ML training run.

## What I built

A FastAPI backend parses the query with Groq (structured extraction: cuisine preference, price tier, location, occasion signals), filters the dataset, and re-ranks candidates by a weighted score. A Next.js frontend streams results with a confidence explanation per recommendation.

## Outcome & learning

Sub-2-second latency end-to-end, which required moving the embedding step from OpenAI to Groq. The pipeline handles compound queries well but struggles with implicit preferences ("budget-friendly" means different things by locality). A user preference profile would fix this — useful for v2.
